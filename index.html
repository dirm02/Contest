Shared Database, Isolated Schemas

All four modules share the same PostgreSQL database on Render. Each uses its own schema (cra, fed, ab, general) so tables never collide. Cross-dataset queries use schema-qualified names like cra.cra_identification or fed.grants_contributions.

The Fuzzy Matching Challenge

The defining challenge across these datasets is entity resolution. The same organization can appear under dozens of name variations across CRA, FED, and AB — including trade names, bilingual French-English pairs, typos, and the multiple Business Number suffix variants each registered entity gets (RR for the charity account, RC for corporate tax, RP for payroll). A typical mid-sized multi-dataset charity carries 10+ distinct name variants in the source data. The general module provides a multi-layer resolution engine with AI-assisted review.

Quick Start Guide

Prerequisites

Node.js 18+ required. All data is already loaded in the shared database — you only need to install dependencies to start querying and running analysis.

1

Clone and Install

git clone <repo-url> && cd hackathon
for dir in CRA FED AB general; do (cd $dir && npm install); done
2

Verify Database Access

Each module ships with .env.public containing read-only credentials. Test connectivity:

cd CRA && npm run verify
cd ../FED && npm run verify
cd ../AB && npm run verify
3

Configure API Keys (optional, for AI features)

To use AI-assisted entity resolution, add credentials to general/.env.public or your personal general/.env:

# Option A: Anthropic Direct API
ANTHROPIC_API_KEY=sk-ant-api03-your-key-here

# Option B: Vertex AI (Google Cloud)
VERTEX_SERVICE_ACCOUNT_JSON={"type":"service_account",...}
VERTEX_PROJECT_ID=your-project-id
VERTEX_CLAUDE_SONNET_MODEL=claude-sonnet-4-6
VERTEX_ENDPOINT=aiplatform.googleapis.com
VERTEX_LOCATION_ID=global
VERTEX_METHOD=rawPredict

The system tries Anthropic first, falls back to Vertex automatically. You can force a provider with --provider vertex.

4

Run Analysis

# CRA: Detect circular gifting patterns
cd CRA && npm run analyze:all

# AB: Run all 6 advanced analysis scripts
cd ../AB && npm run analyze:all

# General: Run the full entity-resolution pipeline (CRA + FED + AB)
cd ../general
npm run entities:dashboard        # browser dashboard at localhost:3800
npm run entities:dossier          # per-entity dossier explorer at localhost:3801
5

Adapt to Your Own LLM

The AI review module (general/lib/llm-review.js) has a clean provider abstraction. To add a new provider:

  1. Add a callYourProvider(prompt, options) function returning { text, usage }
  2. Register it in availableProviders() by checking your env variable
  3. Add it to the fallback chain in callLLM()

The prompt and response parsing are provider-agnostic — any LLM that returns JSON works.

Environment Loading Order

All modules load environment variables consistently:

  1. .env.public loaded first (shared defaults for hackathon participants)
  2. .env loaded second with override: true (personal overrides win)

Participants without a .env get the shared read-only credentials. Developers with a .env containing admin credentials automatically override for write operations (migrations, imports, drops).

General — Cross-Dataset Entity Resolution

Schema: general  |  Cross-dataset utilities, reference data, and the entity resolution pipeline.

The Problem

The same legal organization appears under dozens of name variations across CRA, federal, and Alberta sources. A representative mid-sized multi-dataset charity typically has 10+ distinct name variants in the source data (legal names, trade names, typos, bilingual French-English pairs), carries multiple Business Number suffix variants (RR for the charity account, RC for corporate tax, RP for payroll), and contributes 200+ source records across 6 different tables. Without reconciling all of these to one canonical entity, cross-dataset accountability analysis is impossible.

The general module produces one golden record per real-world organization, linked to every source row that contributed to it.

The Three Techniques

The pipeline combines three complementary approaches to entity matching. Each catches cases the others miss.

1. Deterministic matching (rules-based)

Walks the six source tables in trust order — CRA first (every registered charity has a Business Number), then federal grants, then Alberta data — and applies a cascade of matching rules. BN-based matching handles the gold-standard cases; normalized-name matching strips out operational noise like TRADE NAME OF, O/A, DBA, AKA, FORMERLY, and bilingual English-French pipe separators; trade-name extraction catches the common Canadian pattern where an organization operates under a different trading name.

2. Probabilistic matching (Splink)

Integrates Splink, the UK Ministry of Justice's open-source Fellegi-Sunter record linkage library. Splink weights multiple features — name similarity (Jaro-Winkler at thresholds 0.92 and 0.82), city, postal code, entity type, and BN — and learns the relative weights from the data itself through expectation-maximization training. Runs on a DuckDB backend for speed (~3 minutes on ~1 million records), then writes pairwise match probabilities and connected-component cluster IDs back to Postgres.

Splink catches the cases that rules miss: hierarchical organizations where word order differs, truncated federal-registry variants, no-BN cross-dataset pairs that match by address rather than name. It is fully deterministic given a fixed seed — two runs produce bit-identical predictions.

3. LLM verdict and canonical-name authoring

Claude Sonnet 4.6 reviews every candidate pair produced by the deterministic and probabilistic stages. 100 concurrent workers per provider, running against both Anthropic's direct API and Google Vertex AI in parallel (separate rate-limit pools). Each LLM call does two things in one pass:

  • Decide whether the two entities are the SAME legal organization, RELATED (parent/subsidiary/regional chapter/affiliate), or DIFFERENT.
  • Author the golden record for SAME verdicts: the canonical legal name (the cleanest, most official-looking form chosen from all observed variants), the entity type (charity / non-profit / company / government / grant-recipient / individual / unknown), and an exhaustive deduplicated list of every meaningful name variant across both entities.

LLM review is resumable across crashes. Workers claim candidates via row-level locks; stuck reviews are reset automatically after 10 minutes. 429 rate-limit errors trigger 30-120 second exponential backoff.

Why CRA Qualified Donees Matter

Every registered Canadian charity declares its gifts to other charities on its annual T3010 filing, writing out the recipient's name as it knows it. This produces an under-appreciated gold mine: the same organization appears under many different names written by many different donors — truncations, program-prefixed variants (e.g. "[PROGRAM NAME] — [ORG NAME]"), and line-item numbers that are often idiosyncratic to the filer's accounting. For any given charity, 5–10 additional name variants typically surface from qualified-donee declarations that don't appear in any other source. The pipeline pulls these in explicitly as a dedicated sub-phase.

Design Principles

  • BN is the primary identifier. Every stage treats the 9-digit Canadian Business Number root as authoritative. Where it's available it overrides name evidence; where it's missing we cascade to weaker signals.
  • Every stage is idempotent and resumable. Unique constraints, ON CONFLICT DO NOTHING on inserts, and row-level locks on LLM claims mean re-running any phase is safe and continues from where it left off.
  • Observable by default. The dashboard polls the database directly, not a separate event stream. What you see is the actual state of the pipeline, always.
  • No destructive operations without explicit consent. The reset script requires --yes or a dashboard confirmation dialog.
